NestJS Module Circular Reference — Dependency Injection Fails at Startup
nestjs di-container
Fixed ▶
🚨 Error
Nest cannot create the LeadsModule instance. The module at index [1] of the LeadsModule "imports" array is undefined. Potential causes: - A circular dependency between modules. Use forwardRef() to avoid it. - Type: undefined Error: Nest cannot create the LeadsModule instance. at InstanceLoader.createInstancesOfModules (node_modules/@nestjs/core/injector/instance-loader.js:86:19) at node_modules/@nestjs/core/injector/instance-loader.js:55:13 [Nest] 12034 - 04/01/2026, 09:14:22 AM ERROR [ExceptionHandler] Nest cannot create the LeadsModule instance. The module at index [1] of the LeadsModule "imports" array is undefined.
⚠️ Root Cause
When setting up the project, I forgot to export CoreModule from AppModule before importing it in feature modules. The import order in the module registry also triggered a circular reference: AutomationModule imported LeadsModule, and LeadsModule imported ActivitiesModule, which imported AutomationModule. NestJS resolves this lazily, but because the export array was incomplete, the DI container saw undefined at index [1].

Additionally, @Global() was declared on CoreModule but it was not being imported in AppModule.imports — only in child modules — meaning the global scope was never registered.
🛠 Fix

@Module({
  imports: [
    // CoreModule MUST be first — it's @Global() and exports PrismaService, RedisService, WsService
    CoreModule,
    ConfigModule.forRoot({ isGlobal: true, validate: validateEnv }),
    ThrottlerModule.forRootAsync({ useFactory: () => [{ ttl: 60000, limit: 200 }] }),
    BullModule.forRootAsync({ useFactory: (config) => ({ connection: { url: config.get('REDIS_URL') } }), inject: [ConfigService] }),

    // Feature modules — all resolved AFTER CoreModule is globally available
    AuthModule,
    LeadsModule,
    ContactsModule,
    DealsModule,
    AutomationModule,   // circular: wrap with forwardRef() where needed
    ActivitiesModule,
    // ...
  ],
})
export class AppModule {}

// core.module.ts — mark @Global and export all shared services
@Global()
@Module({
  providers: [PrismaService, RedisService, WsService, TenantContextService],
  exports:    [PrismaService, RedisService, WsService, TenantContextService],
})
export class CoreModule {}
🧠 Post-Mortem
Key rule: In NestJS, a module decorated with @Global() must still be imported once at the root AppModule. The @Global() decorator only means "once registered, providers are visible everywhere" — it does NOT self-register the module. Also, circular deps between feature modules must use forwardRef() and are a strong code smell indicating the modules have overlapping responsibilities that should be refactored.

#003
Prisma $use() Middleware Deprecated — Type Error on params.model
prisma postgres
Fixed ▶
🚨 Error
TypeError: this.prisma.$use is not a function at PrismaService.onModuleInit (src/core/prisma.service.ts:24:16) at NestFactory.createApplicationContext # Also encountered during migration to Prisma v5: error TS2339: Property '$use' does not exist on type 'PrismaClient' # The Prisma docs for v4 showed $use() but v5 replaced it with $extends()
⚠️ Root Cause
Prisma v5 removed the $use() middleware API and replaced it with the Client Extensions API ($extends()). The old middleware signature took a params object with model, action, args, and a next() function. The new extension system is fully typed and composable but has a completely different API surface. All tenant-scoping middleware had to be rewritten.
🔁 Diff
prisma.service.ts — $use → $extends migration
Before (Prisma v4)
this.prisma.$use(async (params, next) => { const tenantId = this.ctx.get(); const skip = ['Tenant','BillingInfo']; if (tenantId && !skip.includes(params.model)) { if (!params.args.where) params.args.where = {}; params.args.where.tenantId = tenantId; } return next(params); });
After (Prisma v5)
this.prisma = this.prisma.$extends({ query: { $allModels: { async $allOperations({ model, operation, args, query }) { const tenantId = ctx.getStore()?.tenantId; const skip = ['Tenant', 'BillingInfo']; if (tenantId && !skip.includes(model)) { args.where = { ...args.where, tenantId }; } return query(args); }, }, }, }) as PrismaClient;
🧠 Post-Mortem
Lesson: Prisma's extension API is more powerful but more verbose. The key insight was that $extends() returns a new extended client, not mutating the original — so the result must be reassigned. Also, the extended client's TypeScript type changes, requiring an explicit cast in some cases. Always check the Prisma changelog when upgrading major versions — the $use deprecation was a breaking change silently introduced in v5.0.0.

Prisma Decimal Type Mismatch — deal.value Returns String Instead of Number
prisma postgres
Fixed ▶
🚨 Error
// Frontend received deal.value as string "125000.00" instead of number // Analytics revenue calculation: "125000.00" + "75000.00" = "125000.0075000.00" 💀 TypeError: Cannot perform arithmetic on type 'string' at AnalyticsService.calculateRevenue (src/analytics/analytics.service.ts:87:28) // In the Deal response: { "value": "125000.00", "probability": "0.65" } // ❌ strings { "value": 125000.00, "probability": 0.65 } // ✅ expected numbers
⚠️ Root Cause
Prisma maps PostgreSQL DECIMAL/NUMERIC columns to its own Decimal class (from decimal.js), not JavaScript number. When serialized to JSON via JSON.stringify(), Decimal instances serialize as strings, not numbers. This caused silent data corruption in the analytics aggregation service where revenue totals were being string-concatenated rather than numerically summed. The bug was especially nasty because TypeScript typed the value correctly as Prisma.Decimal, but the serialization was silently wrong.
🛠 Fix
typescript — src/core/prisma.service.ts (serialization fix) 
// Option 1: Convert Decimal to number in the repository layer
async findAllDeals(tenantId: string): Promise<DealResponseDto[]> {
  const deals = await this.prisma.deal.findMany({ where: { tenantId } });
  return deals.map(d => ({
    ...d,
    value: Number(d.value),          // ✅ Decimal → number
    probability: d.stage ? Number(d.stage.probability) : null,
  }));
}

// Option 2: Global serialization in ResponseTransformInterceptor
function serializeDecimals(obj: unknown): unknown {
  if (obj instanceof Prisma.Decimal) return Number(obj);
  if (Array.isArray(obj)) return obj.map(serializeDecimals);
  if (obj !== null && typeof obj === 'object') {
    return Object.fromEntries(
      Object.entries(obj).map(([k, v]) => [k, serializeDecimals(v)])
    );
  }
  return obj;
}

// analytics.service.ts — never do string arithmetic on Decimal
const revenue = deals.reduce((sum, d) => sum + Number(d.value), 0); // ✅
🧠 Post-Mortem
Prisma's Decimal type is NOT a JS number. It's a decimal.js instance designed for precision arithmetic. Always call Number() or .toNumber() before doing math or serializing to JSON. The safest fix is a global response interceptor that recursively converts all Decimal instances on the way out. This prevents the bug from silently creeping in again as new endpoints are added.

#006
Prisma Transaction Deadlock — Pipeline Stage Reorder Creates Circular Wait
prisma postgres
Fixed ▶
🚨 Error
PrismaClientKnownRequestError: Transaction failed due to a write conflict or a deadlock. Please retry your transaction. Code: P2034 at PrismaService.reorderStages (src/pipelines/pipeline.repository.ts:112:7) # Triggered when two users drag stages simultaneously on the Kanban pipeline # Intermittent — happens ~1 in 20 concurrent reorder operations
⚠️ Root Cause
The reorderStages function executed multiple sequential UPDATE statements inside a Prisma transaction, one per stage, in the order the client sent them. When two concurrent requests reordered the same pipeline simultaneously, they would acquire row locks in different orders — creating a classic deadlock cycle. User A locked Stage 1 then tried to lock Stage 2; User B locked Stage 2 and tried to lock Stage 1. Postgres detected the cycle and aborted one transaction.
🛠 Fix
typescript — src/pipelines/pipeline.repository.ts 
>async reorderStages(
  pipelineId: string,
  stageOrders: Array<{ id: string; position: number }>,
): Promise<void> {
  // ✅ FIX 1: Sort by ID to guarantee consistent lock acquisition order
  // This prevents deadlock by ensuring all transactions lock rows in the same sequence
  const sorted = [...stageOrders].sort((a, b) => a.id.localeCompare(b.id));

  await this.prisma.$transaction(
    sorted.map(({ id, position }) =>
      this.prisma.stage.update({
        where: { id, pipelineId },  // ✅ pipelineId in where ensures we own this stage
        data:  { position },
      })
    ),
    {
      isolationLevel: Prisma.TransactionIsolationLevel.ReadCommitted, // ✅ FIX 2
      timeout: 5000, // ✅ FIX 3: never hang forever
    }
  );
}

// ✅ FIX 4: Retry on P2034 at the service layer
async reorderStagesWithRetry(...args: Parameters<typeof this.reorderStages>) {
  for (let i = 0; i < 3; i++) {
    try { return await this.reorderStages(...args); }
    catch (e) {
      if (e.code === 'P2034' && i < 2) { await sleep(50 * (i + 1)); continue; }
      throw e;
    }
  }
}
🧠 Post-Mortem
Deadlock prevention requires consistent lock ordering. The canonical fix is to sort rows by primary key before locking them — this guarantees all concurrent transactions acquire locks in the same sequence, making a deadlock cycle impossible. Combine with retry logic (exponential backoff, max 3 attempts) because even with sorted locking, high-concurrency scenarios can still trigger timeouts. ReadCommitted isolation is sufficient for this use case and reduces contention vs. Serializable.

typescript — src/contacts/contacts.service.ts 
>async deleteContact(id: string): Promise<void> {
  // ✅ Use a transaction to delete all related data atomically
  await this.prisma.$transaction([
    // Polymorphic relations — must be deleted manually (no DB FK cascade)
    this.prisma.activity.deleteMany({
      where: { entityType: 'CONTACT', entityId: id }
    }),
    this.prisma.task.deleteMany({
      where: { entityType: 'CONTACT', entityId: id }
    }),
    // Real FK relations — Prisma cascade handles these if schema is correct
    this.prisma.communication.deleteMany({ where: { contactId: id } }),
    this.prisma.ticketReply.deleteMany({
      where: { ticket: { contactId: id } }
    }),
    this.prisma.ticket.deleteMany({ where: { contactId: id } }),
    // Finally delete the contact itself
    this.prisma.contact.delete({ where: { id } }),
  ]);

  await this.redis.invalidateByPattern(`contacts:*`);
  this.wsService.emitToTenant('contact.deleted', { id });
}

// schema.prisma: ensure real FK relations have explicit cascade
model Communication {
  contact Contact @relation(fields: [contactId], references: [id], onDelete: Cascade)
}
🧠 Post-Mortem
Polymorphic relations require manual cleanup. Since activities and tasks reference entities by string ID (not a real FK), Postgres has no way to cascade-delete them. The service layer owns this responsibility. Use a transaction to make deletion atomic — if any step fails, nothing is deleted. Added a DB maintenance job that runs weekly to detect and log any orphan records as a safety net.

#008
Multi-Tenancy Data Leakage — AsyncLocalStorage Context Leaks Between Concurrent Requests
multi-tenant nestjs security
Critical ▶
🚨 Error
// Tenant A (acme.com) sent a GET /api/leads request // Tenant B (globex.com) sent a simultaneous GET /api/leads request // Under high concurrency, Tenant B's request returned Tenant A's leads! // Log snippet (load test at 500 RPS): [SECURITY] tenantId mismatch detected: request=globex_tenant_id, result_tenantId=acme_tenant_id, endpoint=GET /api/leads userId=user_xyz, count=3 // Root cause traced to: TenantContextMiddleware not using AsyncLocalStorage correctly // Context was being stored as a module-level variable, not per-request store
⚠️ Root Cause
The original implementation stored tenantId in a module-level singleton (a plain class field on TenantContextService). This worked fine for single-threaded sequential requests but failed catastrophically under concurrency: Request B would overwrite the singleton with its tenantId while Request A's async chain was still executing. The fix required using Node.js AsyncLocalStorage — which propagates per-request context automatically through all async callbacks and await chains.
🛠 Fix
typescript — src/core/tenant-context.service.ts 
>import { AsyncLocalStorage } from 'async_hooks';

interface TenantStore {
  tenantId: string;
  userId: string;
  userRole: UserRole;
}

// ✅ Module-level ALS — one instance shared across all requests
// Each request gets its OWN isolated store via als.run(store, callback)
const als = new AsyncLocalStorage<TenantStore>();

@Injectable()
export class TenantContextService {
  run(store: TenantStore, callback: () => void) {
    return als.run(store, callback);  // ✅ scoped to this request's async tree
  }
  get(): TenantStore | undefined { return als.getStore(); }
  getTenantId(): string {
    const store = this.get();
    if (!store) throw new Error('No tenant context — is middleware running?');
    return store.tenantId;
  }
}

// middleware: wrap each request in als.run() BEFORE handing to NestJS
@Injectable()
export class TenantContextMiddleware implements NestMiddleware {
  use(req, res, next) {
    const payload = this.jwtService.decode(req.headers.authorization?.split(' ')[1]);
    const store: TenantStore = {
      tenantId: payload?.tenantId,
      userId:   payload?.sub,
      userRole: payload?.role,
    };
    // ✅ next() is called INSIDE als.run() — entire NestJS handler chain
    // runs within this store's async scope
    this.ctx.run(store, () => next());
  }
}
🧠 Post-Mortem
This was the most critical bug in the entire project. Serving one tenant's data to another is a catastrophic SaaS failure. The root cause was treating Node.js like a single-threaded synchronous runtime — which it is for CPU, but NOT for async I/O. Multiple requests can have their async chains interleaved concurrently.
  • AsyncLocalStorage is the Node.js idiomatic solution — analogous to Java's ThreadLocal but for async contexts
  • The middleware must call next() inside the als.run() callback, not before or after
  • Added a validation layer: the Prisma extension double-checks the returned rows' tenantId matches the store

AsyncLocalStorage Returns undefined Inside BullMQ Workers
multi-tenant bullmq nestjs
Fixed ▶
🚨 Error
Error: No tenant context — is middleware running? at TenantContextService.getTenantId (src/core/tenant-context.service.ts:18:13) at PrismaService.$extends.query.$allModels.$allOperations at EmailWorker.process (src/jobs/workers/email.worker.ts:34:19) // Workers process jobs from the Redis queue — there's NO HTTP request, // so TenantContextMiddleware never ran, and AsyncLocalStorage is empty.

BullMQ workers run in a completely separate async context — they are not part of any HTTP request lifecycle.     private readonly ctx: TenantContextService,

    await new Promise<void>((resolve, reject) =>

The als.run() wrapper approach works but can mask missing explicit scoping in nested calls. Standardized on passing tenantId in every worker query.
#010
WebSocket Events Leaking Across Tenants — Missing Tenant Room Scoping
websocket multi-tenant security
Fixed ▶
🚨 Error
// During integration testing — two browser sessions, different tenants: // Session A: admin@acme.com tenantId: tenant_acme // Session B: admin@globex.com tenantId: tenant_globex // When Acme creates a new lead: // Session A receives: { event: 'lead.created', data: { id: 'lead_xyz', ...acmeData } } // Session B ALSO receives: { event: 'lead.created', data: { id: 'lead_xyz', ...acmeData } } // ↑ DATA LEAK — Globex sees Acme's events // Root cause: WsGateway was broadcasting to ALL connected sockets, not tenant rooms

🛠 Fix
typescript — src/core/websocket/ws.gateway.ts 
>@WebSocketGateway({ cors: { origin: process.env.CORS_ORIGINS.split(',') } })
export class WsGateway implements OnGatewayInit, OnGatewayConnection {
  @WebSocketServer() server: Server;

  afterInit(server: Server) {
    server.use((socket, next) => {
      const token = socket.handshake.auth.token;
      try {
        const payload = this.jwtService.verify<JwtPayload>(token);
        socket.data.tenantId = payload.tenantId;
        socket.data.userId   = payload.sub;
        next();
      } catch { next(new Error('Unauthorized')); }
    });
  }

  async handleConnection(socket: Socket) {
    const { tenantId, userId } = socket.data;
    // ✅ Join tenant room + personal room
    await socket.join(`tenant:${tenantId}`);
    await socket.join(`user:${userId}`);
    socket.emit('connected', { userId, tenantId });
  }
}

// ws.service.ts — scoped emit methods
@Injectable()
export class WsService {
  emitToTenant(tenantId: string, event: string, data: unknown) {
    // ✅ Emit ONLY to sockets in this tenant's room
    this.gateway.server.to(`tenant:${tenantId}`).emit(event, data);
  }
  emitToUser(userId: string, event: string, data: unknown) {
    this.gateway.server.to(`user:${userId}`).emit(event, data);
  }
}
🧠 Post-Mortem
Never use server.emit() in a multi-tenant system. This broadcasts to ALL connected clients globally. Always scope to a room. Socket.io rooms are cheap, stateful, and automatically cleaned up on disconnect. The fix also added a connection audit log to track which users are connected to which tenant rooms, enabling real-time presence detection.
#011
Race Condition on Deal Stage Move — Concurrent Moves Return Stale stageId
prisma postgres concurrency
Fixed ▶
🚨 Error
// Two users drag the same deal simultaneously on Kanban: // User A moves deal_123: Stage "Proposal" → "Negotiation" // User B moves deal_123: Stage "Proposal" → "Closed Won" // Expected: Last write wins, deal ends in "Closed Won" // Actual: Deal ends in "Negotiation", stage history shows both moves // But kanban re-renders showing "Closed Won" on User B's screen // and "Negotiation" on User A's screen → inconsistent UI // Backend logs: [LeadsService] deal_123 moved to stage_negotiation by user_A at 14:32:01.445 [LeadsService] deal_123 moved to stage_closed_won by user_B at 14:32:01.447 // 2ms apart — both passed the validation check before either committed
⚠️ Root Cause
The moveStage service method was doing a read-check-write pattern without any locking: it read the current stageId, validated the move was valid, then wrote the new stageId. Two concurrent requests could both pass the read phase and both perform the write — a classic TOCTOU (Time-Of-Check-Time-Of-Use) race condition. >@Injectable()
export class RedisService {
  private isConnected = true;

  // ✅ Fault-tolerant cache-aside pattern
  async cacheOrFetch<T>(
    key: string,
    fetcher: () => Promise<T>,
    ttlSeconds: number,
  ): Promise<T> {
    if (this.isConnected) {
      try {
        const cached = await this.client.get(key);
        if (cached) return JSON.parse(cached);
      } catch (e) {
        // ✅ Cache miss is not an error — log and fall through to DB
        this.logger.warn(`Cache read failed for key ${key}: ${e.message}`);
        this.isConnected = false;
        // Schedule reconnect check
        setTimeout(() => this.checkConnection(), 5000);
      }

    // ✅ Always falls back to DB — API stays up even if Redis is down
    const data = await fetcher();

    if (this.isConnected) {
      // Try to cache — don't throw if it fails
      this.client.setex(key, ttlSeconds, JSON.stringify(data)).catch(() => {});
    }
    return data;
  }

    try {
      await this.client.ping();
      this.isConnected = true;
      this.logger.log('Redis reconnected');
    } catch { setTimeout(() => this.checkConnection(), 10000); }
  }
}

🚨 Error

The refresh token rotation flow was: (1) verify token hash, (2) generate new tokens, (3) update DB with new hash. 🛠 Fix
typescript — src/auth/auth.service.ts (atomic rotation) 
>async refreshTokens(userId: string, incomingToken: string): Promise<TokenPair> {
  return this.prisma.$transaction(async (tx) => {
    // ✅ Atomic: lock the user row, verify hash, rotate — in one transaction
    const user = await tx.$queryRaw`
      SELECT id, email, role, refresh_token_hash
      FROM users WHERE id = ${userId}
      FOR UPDATE  -- ✅ row-level lock prevents concurrent rotations
    `;

    if (!user[0]?.refreshTokenHash) throw new UnauthorizedException();

    const isValid = await bcrypt.compare(incomingToken, user[0].refreshTokenHash);
    if (!isValid) {
      // ✅ REUSE DETECTED: token already rotated — invalidate ALL sessions
      await tx.user.update({ where: { id: userId }, data: { refreshTokenHash: null } });
      throw new UnauthorizedException('Token reuse detected — all sessions revoked');
    }

    const { accessToken, refreshToken } = this.generateTokenPair(user[0]);
    const hash = await bcrypt.hash(refreshToken, 10);
    await tx.user.update({ where: { id: userId }, data: { refreshTokenHash: hash } });
    return { accessToken, refreshToken };
  }, { isolationLevel: Prisma.TransactionIsolationLevel.Serializable });
}

This limits the blast radius of a stolen token — the attacker loses access the moment the legitimate user refreshes. Also switched refresh token storage to httpOnly cookies to prevent XSS extraction.
